### 本次使用 AI 技术
- 本产品本身不使用 LLM
- 唯一专用 AI 模型：Spotify Basic Pitch（音频转音符）
- Claude Code 辅助完成：产品方案 brainstorming、技术选型对比分析、设计文档撰写

### 开发落地难点
- **琵琶音色获取**：免费且可商用的高质量琵琶 SoundFont 难以找到，最终选择 FluidR3 GM（MIT 许可），后续迭代自行录制或合作获取更好音源
- **技术选型的取舍**：LLM 编曲 vs 规则引擎、服务端 vs 浏览器端、Canvas vs SVG vs HTML，每个决策点都需要在成本/性能/效果三角中权衡
- **六维度难度定义的确定性**：需要将模糊的「难度」概念精确到可量化的算法参数

### 解决方案与取舍
- **放弃 LLM 编曲**：六维度均可量化为确定性规则，LLM 引入不必要的成本和不确定性。架构预留插件接口，未来可扩展
- **放弃真实琵琶采样**：免费商用采样找不到，决定 MVP 用 SoundFont 跑通链路，后续迭代升级音源
- **选择浏览器端全链路**：零服务端成本、离线可用是差异化亮点，虽然首次加载体积较大（~30MB）
- **选择 SVG+Canvas 双层渲染**：Canvas 负责古风纹理（静态），SVG 负责音符交互（动态），各取所长

### 产品复盘思考
- 琵琶谱生成这个需求，技术上的核心瓶颈是「音频转音符」而非「编曲」，因为编曲规则可以工程师手工定义，但音频识别是纯 AI 能力。Basic Pitch 对琵琶音色的识别效果需要实际验证
- 零 LLM 零 API 的路线让产品完全离线可用，这在面试中是非常好的技术亮点：「用最朴素的算法解决最专业的音乐问题」
- 预置素材库的思路调整（琵琶乐 → 转谱素材+经典参考）是一次重要的产品判断：输入和输出不能是同一个东西，产品的核心价值是「把非琵琶曲转成琵琶谱」
- 后续最值得关注的方向：多乐器支持（古筝/二胡），因为编曲引擎的六维度框架可以复用到其他民族乐器
