### 本次改动核心功能

- 完成古风 UI 首页（水墨动画背景 + 拖拽上传 + 预置素材快捷入口）

- 完成 SVG + Canvas 双层谱面渲染系统（宣纸纹理背景 + 简谱数字音符 + 技法符号）
- 完成 SoundFont 琵琶音色播放器（播放/暂停/调速 + 进度条）
- 完成三难度标签切换（入门/进阶/专业，含难度描述）
- 完成曲谱导出功能（JSON 数据导出 + 浏览器打印）
- 完成曲谱库页面（LocalStorage 持久化 + 历史记录查看）
- 完成预置素材库（20 首曲目：10 首转谱素材 + 10 首琵琶经典参考）

- 更新产品介绍.md 全部功能状态为「已实现」，版本号更新至 v1.0 MVP

### 本次使用 AI 技术
- Spotify Basic Pitch（音频转 MIDI 音符，浏览器端 tfjs 推理）
- 纯规则引擎编曲（六维度量化参数，确定性输出，零 LLM 依赖）
- Web Audio API + SoundFont（MIDI → 琵琶音色合成）
- SVG + Canvas 双层混排（Canvas 古风纹理 + SVG 交互音符）
- Next.js 全栈框架 + Tailwind CSS 样式

### 开发落地难点
- **全链路浏览器端运行**：ffmpeg.wasm、Basic Pitch、SoundFont 三个重型依赖首次加载约 30MB，需优化加载策略和进度反馈
- **简谱记谱法映射**：MIDI 12 音高体系与简谱 7 音体系之间的映射规则，需处理升降号、调性转换等边界
- **SoundFont 精简**：完整 FluidR3 GM SoundFont 约 140MB，需精简为仅琵琶音色的 ~5MB
- **SVG 谱面布局**：简谱数字音符 + 技法符号 + 弦位品位的空间排布，需保证不同密度的谱面都能清晰可读

### 解决方案与取舍
- **放弃 LLM 编曲**：六维度难度控制全部量化为规则引擎参数，确定性输出保证同一输入每次生成相同结果，且零 API 成本
- **放弃真实琵琶采样**：免费可商用高质量采样难找，SoundFont 跑通链路，后续迭代可升级音源
- **放弃服务端处理**：全链路浏览器端运行，零服务端成本，离线可用是差异化亮点
- **选择 SVG + Canvas 双层渲染**：Canvas 负责静态装饰（宣纸纹理），SVG 负责交互内容（音符高亮、点击试听），各取所长

### 产品复盘思考
- v1.0 MVP 证明了一个核心假设：非音乐专业的开发者可以用算法实现琵琶谱自动生成，且全链路浏览器端零成本运行
- 六维度难度控制框架是可复用的产品资产，后续扩展到古筝、二胡等民族乐器时只需调整音域、技法表和把位规则
- 零 LLM 零 API 的架构路线在面试中是很好的差异化亮点：「用最朴素的算法解决最专业的音乐问题」
- v1.1 优先方向：播放与谱面联动高亮跟弹、节拍器功能、移动端响应式适配

## v0.1 (3) — 2026-05-23 — 新增：谱面渲染系统（SVG + Canvas + 播放器 + 谱面页）

### 迭代基础信息
- **更新时间**：2026-05-23
- **版本号**：v0.1 (3)
- **迭代类型**：新增

### 本次改动核心功能
- 新增 `ScoreRenderer` 主组件，整合谱面渲染全链路（Canvas 背景 + SVG 音符 + 难度切换 + 播放控制 + 导出）
- 新增 `CanvasBackground`：Canvas 绘制古风纸纹背景（水墨纹理 + 径向渐变），不参与音符交互
- 新增 `SvgScore`：SVG 渲染简谱数字音符，支持技法符号标注、弦位品位置标注、时值底线显示
- 新增 `DifficultyTabs`：三级难度切换控件（入门/进阶/专业），含难度描述文案
- 新增 `PlayerControls`：播放/暂停/停止 + 速度调节滑块 + 进度条，集成 SoundFont 实时合成播放
- 新增 `ExportPanel`：导出 JSON 谱面数据 + 浏览器打印功能
- 新增 `/score/[id]` 路由页，从 sessionStorage 加载三难度谱面数据

### 本次使用 AI 技术
- 无 LLM 调用，纯前端组件开发
- SVG 渲染采用简谱数字记谱法（1-7 对应音名），适配中国民乐谱面习惯

### 开发落地难点
- Canvas 与 SVG 的 z-index 分层：Canvas 负责静态装饰（position: absolute, z-0），SVG 负责交互内容（z-10），互不干扰
- 简谱显示逻辑：将 12 个 MIDI pitch class 映射为 7 个简谱数字（1-7），升/降音通过相对音高关系隐含表达

### 解决方案与取舍
- 选择 SVG 而非纯 Canvas 渲染音符：SVG 天然支持缩放、text 元素可被屏幕阅读器识别、元素可单独绑定事件
- 简谱不显式标升降号：琵琶谱面传统以指法为准，音名作为辅助信息标注在弦位/品位上

### 产品复盘思考
- 谱面渲染是产品的「最终交付物」，直接决定用户是否愿意使用。SVG+Canvas 双层方案兼顾视觉美感与功能交互，是面试中值得展开的技术决策点
- 当前版本谱面为静态 SVG，后续可加注拍号、小节线、反复记号等专业乐谱元素

### 本次改动核心功能
- 创建 `src/audio/basicPitchTranscriber.ts`，封装 Spotify Basic Pitch 的音频转 MIDI 流程
- 提供三个导出函数：`initBasicPitch`（延迟加载 TF 模型）、`transcribe`（音频 → RawNote[]）、`validateHasMelody`（是否包含有效旋律）
- 实时进度回调机制，支持 UI 反馈加载/推理进度

### 本次使用 AI 技术
- 无（纯 TypeScript 封装层，适配 @spotify/basic-pitch 的 evaluateModel + outputToNotesPoly 真实 API）

### 开发落地难点
- @spotify/basic-pitch 的 API 是回调式的（evaluateModel 通过 onComplete 累积帧数据），需要封装为 Promise 接口才能接入编曲引擎流程
- 模型文件（TF GraphModel）需在构建时从 node_modules 复制到 public/ 目录供浏览器加载

### 解决方案与取舍
- 将 evaluateModel 的逐窗口回调累积到 frames/onsets/contours 数组中，待全部完成后统一调用 outputToNotesPoly 转为音符事件
- 模型路径硬编码为 `/models/basic-pitch/model.json`，后续可通过环境变量配置

### 产品复盘思考
- Basic Pitch 的逐窗口回调设计实际上有利于显示细粒度进度（0-100%），比单一 Promise 返回对用户体验更友好
- validateHasMelody 的阈值（5 个不同音高 + 10 个音符）是经验值，后续需要根据实际测试数据校准

## v0.1 — 2026-05-22 — 新增：产品方案设计与技术选型完成

### 迭代基础信息
- **更新时间**：2026-05-22
- **版本号**：v0.1
- **迭代类型**：新增（项目启动）

### 本次改动核心功能
- 完成产品定位和用户需求分析（目标用户：琵琶学习者）
- 确定技术方案：Next.js + ffmpeg.wasm + Basic Pitch + 规则引擎 + SVG/Canvas + SoundFont
- 确定全链路浏览器端处理、零 LLM 依赖、零外部 API 调用的架构路线
- 设计六维度难度控制系统（音符密度、节奏、技法种类、技法频率、把位、速度）
- 设计预置素材库方案：10 首转谱素材 + 10 首琵琶经典参考
- 设计曲谱库方案：LocalStorage + JSON 导出

### 本次使用 AI 技术
- 本产品本身不使用 LLM
- 唯一专用 AI 模型：Spotify Basic Pitch（音频转音符）
- Claude Code 辅助完成：产品方案 brainstorming、技术选型对比分析、设计文档撰写

### 开发落地难点
- **琵琶音色获取**：免费且可商用的高质量琵琶 SoundFont 难以找到，最终选择 FluidR3 GM（MIT 许可），后续迭代自行录制或合作获取更好音源
- **技术选型的取舍**：LLM 编曲 vs 规则引擎、服务端 vs 浏览器端、Canvas vs SVG vs HTML，每个决策点都需要在成本/性能/效果三角中权衡
- **六维度难度定义的确定性**：需要将模糊的「难度」概念精确到可量化的算法参数

### 解决方案与取舍
- **放弃 LLM 编曲**：六维度均可量化为确定性规则，LLM 引入不必要的成本和不确定性。架构预留插件接口，未来可扩展
- **放弃真实琵琶采样**：免费商用采样找不到，决定 MVP 用 SoundFont 跑通链路，后续迭代升级音源
- **选择浏览器端全链路**：零服务端成本、离线可用是差异化亮点，虽然首次加载体积较大（~30MB）
- **选择 SVG+Canvas 双层渲染**：Canvas 负责古风纹理（静态），SVG 负责音符交互（动态），各取所长

### 产品复盘思考
- 琵琶谱生成这个需求，技术上的核心瓶颈是「音频转音符」而非「编曲」，因为编曲规则可以工程师手工定义，但音频识别是纯 AI 能力。Basic Pitch 对琵琶音色的识别效果需要实际验证
- 零 LLM 零 API 的路线让产品完全离线可用，这在面试中是非常好的技术亮点：「用最朴素的算法解决最专业的音乐问题」
- 预置素材库的思路调整（琵琶乐 → 转谱素材+经典参考）是一次重要的产品判断：输入和输出不能是同一个东西，产品的核心价值是「把非琵琶曲转成琵琶谱」
- 后续最值得关注的方向：多乐器支持（古筝/二胡），因为编曲引擎的六维度框架可以复用到其他民族乐器
